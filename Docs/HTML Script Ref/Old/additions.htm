Here's the old DarkSpace Script Reference for the old API using Lua 4.0 in html and the original doc format.

Here's the list of changes for the new API using Lua 5.0:
Read the whole list first, items may not in any particular order

2003/11/28



clientside:
onFleetSelected( fleetId ) replaces onTeamSelected( teamId )
onShipAttached( shipId ) players ship is spawned, so i.e. entered through a (intersystem) jumpgate, from a shipyard etc
onShipDetached() players ship is removed from the universe, entered a jumpgate / shipyard etc
onDeath() -> onDeath( shipId, killerId ) shipId is the (now destroyed) players ship, killerId the killers Id or nil if no killer exists
onCaptured() -> same modification as onDeath()


flushChat() is currently disabled.
getHex() is currently disabled.
getUnitHex is currently disabled.

playersShip() should be used clientside only

general functions:
canDetect( detectorId, nounId ). 1 if detector can detect the noun, nil if not. Detector may be a ship or planet.
getRadius( nounId ) returns radius ( size ) of the object in gu.
enableTrace( mode ) 0: disable 1: function calls and returns 2: each line 3: calls, returns and each line
trace output ignores enableAlerts() setting. Never use it on live servers, as it will seriously lag the server it is executed on
remoteCall() if specified scope is out of range a global call is made
enableAlert() is a global setting, setting it affects all currently running or to be executed scripts

gadget functions:
isGadgetActive( gadgetId ) - 1 if active, 0 if inactive. The jumpdrive already shows active if its only aligning for a jump.
isJumping() returns if a ship is currently jumping
isGadgetUsable( gadgetId, shiftStatus [, targetId] ) - 1 if true, nil if false.
shiftstatus = 1 if shift should be pressed, 0 if not. TargetId is an optional target.
Some gadgets ( i.e. tractorbeam) need a target for the check, others like ECM not.
If no target is given although needed for the check the function returns nil (not usable)

beamPointDefenceEnabled( gadgetId ) 1 if beam has point defence enabled, nil if not
setBeamPointDefence( gadgetId, enable ) enable = 1 to enable, 0 to disable

getJumpTarget( jumpDriveId ) returns the jumptarget of the jumpdrive. If it has no target nil is returned.

ship functions:
isAIControlled( shipId ) - 1 if true, nil if false. Serverside only !
getShipOrder( shipId ) returns the order for the ship. Numbers are the same as for setOrder()

unit functions:
getUnitExperience( unitId ) returns experience level of the unit ( 0 - 100 ), 0 = green, 91+ = elite
setUnitExperience( unitId, experience ) sets the units experience level


The global names "context" and "ExecutionDepth" are reserved, but since Lua-Scripts aren't permitted to touch them it won't matter anyway

All boolean functions which returned 1/0/nil or 1/nil/nil for Yes/No/Error before now return true/false/nil

These functions now return true if they succeeded, and nil on errors:
setOwner(), enterOrbit(), breakOrbit(), setPosition(), detachNoun(), updateNoun(), startTimer()
sendChat(), planetEvent(), revoltPlanet(), setHull(), orderShip(), setGadgetMount()
useGadget2(), setBeamPointDefence(), setUnitExperience(), addMissionTrigger()
setNextMission(), setMissionState(), pushMission(), pushScoutMission(), pushRepairMission()
pushDefendMission(), pushEscortMission(), pushCaptureMission()

remoteCall(): true only means the call was executed, does not confirm the execution on the target
trapKey() and watchKey(): true: keyHook activated, false: hook was removed, nil: error

Doing things like calling a nil function (mistyped function name), trying to concatenate a nil value etc
outputs an error and immediatly aborts the script execution. Callback functions like timers however could still execute
functions within the script, so make sure starting a timer is the last last thing you do in a function

If any kind of callback fails, i.e. onInitialize(), onTimer(), remoteCall()... the errormessage is only displayed if
enableAlerts(1) was executed previously.

Callbacks ( initiated by remoteCall, startTimer, watchKey etc... ) are "broadcasted" at the destination.
The destination receives the call and executes this in the context of all running scripts.
This allows scripts to communicate, however all defined functions also need unique names or to be
defined within a namespace to prevent "accidental" calls of a function from another script running at the same time
